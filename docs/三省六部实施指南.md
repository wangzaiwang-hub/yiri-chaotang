# 三省六部架构实施指南

## 概述

基于 [OpenClaw 三省六部制](https://github.com/rejecter0116/openclaw) 的设计理念，朝堂游戏已成功实现三省六部协作系统。

## 已完成的工作

### 1. 数据库迁移

创建了 `database/migrate_three_provinces.sql` 迁移脚本：

- 为 `court_members` 表添加 `department` 字段（10个部门）
- 为 `virtual_tasks` 表添加三省六部相关字段：
  - `department` - 负责部门
  - `parent_task_id` - 父任务ID（用于子任务）
  - `plan` - 中书省的规划方案
  - `review_comment` - 门下省的审核意见
  - `sub_tasks` - 子任务列表（JSONB）
- 更新任务状态枚举，支持完整的流转状态
- 创建索引优化查询性能

### 2. 后端实现

#### SecondMe 服务增强 (`backend/src/services/secondme.service.ts`)

- 添加 `getDepartmentPrompt()` 方法，为每个部门定制专属的 AI 角色提示词
- 更新 `executeTask()` 方法，支持传入 `department` 参数

#### 朝堂路由更新 (`backend/src/routes/courts.ts`)

- 创建朝堂时自动设置创建者为皇帝（`department: 'emperor'`）
- 邀请成员时必须指定部门（验证部门有效性）

#### 任务路由重构 (`backend/src/routes/tasks.ts`)

实现完整的三省六部任务流转：

1. **皇帝下旨** (`POST /api/tasks`)
   - 创建任务，状态为 `pending`

2. **中书省规划** (`POST /api/tasks/:id/plan`)
   - 验证规划者是中书省成员
   - 调用 SecondMe AI 生成规划方案
   - 拆解子任务分配给各部门
   - 状态更新为 `reviewing`

3. **门下省审核** (`POST /api/tasks/:id/review`)
   - 验证审核者是门下省成员
   - 调用 SecondMe AI 审核方案
   - 准奏（`approved`）或封驳（`rejected`）

4. **尚书省派发** (`POST /api/tasks/:id/dispatch`)
   - 验证派发者是尚书省成员
   - 状态更新为 `executing`

5. **六部执行** (`POST /api/tasks/:id/execute`)
   - 验证执行者是六部成员之一
   - 调用 SecondMe AI 执行任务（使用部门专属 Prompt）
   - 状态更新为 `reporting`

6. **尚书省汇总** (`POST /api/tasks/:id/report`)
   - 验证汇总者是尚书省成员
   - 调用 SecondMe AI 汇总结果
   - 状态更新为 `completed`

### 3. 前端实现

#### 加入朝堂页面 (`frontend/src/pages/JoinCourt.tsx`)

#### 任务看板页面 (`frontend/src/pages/Tasks.tsx`)

  - 门下省：审核方案

- 显示规划方案、审核意见、执行结果

#### 首页更新 (`frontend/src/pages/Home.tsx`)

- 添加"📋 任务看板"按钮，跳转到任务看板页面

#### API 服务更新 (`frontend/src/services/api.ts`)

## 任务流转状态

```
pending (待规划)
  ↓ 中书省规划
planning (规划中)
  ↓ 提交审核
reviewing (审核中)
  ↓ 门下省审核
  ├─ approved (准奏) → dispatching (派发中) → executing (执行中) → reporting (汇总中) → completed (完成)
  └─ rejected (封驳) → 返回 pending
```

## 使用流程

### 1. 创建朝堂

- 第一个用户创建朝堂，自动成为皇帝（`department: 'emperor'`）

### 2. 邀请好友

- 皇帝生成邀请链接
- 好友通过链接加入，选择加入的部门（三省或六部之一）

### 3. 皇帝下旨

- 皇帝在首页创建任务
- 任务进入 `pending` 状态

### 4. 中书省规划

- 中书省成员在任务看板点击"📜 规划任务"
- SecondMe AI 生成规划方案并拆解子任务
- 任务进入 `reviewing` 状态

### 5. 门下省审核

- 门下省成员在任务看板点击"🔍 审核方案"
- SecondMe AI 审核方案质量
- 选择准奏或封驳

### 6. 尚书省派发

- 如果准奏，尚书省成员点击"📮 派发任务"
- 任务进入 `executing` 状态

### 7. 六部执行

- 六部成员根据自己的专长执行任务
- 点击"⚔️ 执行任务"
- SecondMe AI 使用部门专属 Prompt 完成任务
- 任务进入 `reporting` 状态

### 8. 尚书省汇总

- 尚书省成员点击"📊 汇总回奏"
- SecondMe AI 汇总结果形成奏折
- 任务完成

## 技术亮点

1. **分权制衡**：每个环节都有专门的部门负责，权责清晰
2. **强制审核**：门下省的审核是必经环节，确保方案质量
3. **AI 角色扮演**：每个部门的 SecondMe AI 都有专属的角色提示词
4. **完整审计**：任务的每个环节都有记录（规划、审核、执行、汇总）
5. **实时可见**：任务看板实时显示任务流转状态

## 后续优化方向

4. **实时通知**：任务流转时通知相关部门成员
