# 朝堂·三省六部架构设计

## 概述

参考 [OpenClaw 三省六部制](https://github.com/rejecter0116/openclaw)，将朝堂游戏改造为真正的三省六部协作系统。

## 核心改动

### 1. 角色体系

**原架构：**
- 皇帝（emperor）
- 大臣（minister）

**新架构：**
- 👑 **皇帝**（emperor）- 下旨
- 📜 **中书省**（zhongshu）- 规划任务
- 🔍 **门下省**（menxia）- 审核方案
- 📮 **尚书省**（shangshu）- 派发协调
- 💰 **户部**（hubu）- 数据处理
- 📝 **礼部**（libu）- 文档规范
- ⚔️ **兵部**（bingbu）- 代码实现
- ⚖️ **刑部**（xingbu）- 合规审计
- 🔧 **工部**（gongbu）- 部署运维
- 📋 **吏部**（libu_hr）- 人事管理

### 2. 任务流转

```
皇帝下旨 
  ↓
中书省规划（拆解子任务）
  ↓
门下省审核（准奏/封驳）
  ↓ 准奏
尚书省派发
  ↓
六部并行执行
  ↓
尚书省汇总
  ↓
回奏皇帝
```

### 3. 任务状态

- `pending` - 待中书省规划
- `planning` - 中书省规划中
- `reviewing` - 门下省审核中
- `rejected` - 门下省封驳（需重新规划）
- `approved` - 门下省准奏
- `dispatching` - 尚书省派发中
- `executing` - 六部执行中
- `reporting` - 尚书省汇总中
- `completed` - 已完成
- `failed` - 失败

### 4. 数据库改动

#### 4.1 court_members 表增加部门字段

```sql
ALTER TABLE court_members 
ADD COLUMN department TEXT CHECK (department IN (
  'emperor',      -- 皇帝
  'zhongshu',     -- 中书省
  'menxia',       -- 门下省
  'shangshu',     -- 尚书省
  'hubu',         -- 户部
  'libu',         -- 礼部
  'bingbu',       -- 兵部
  'xingbu',       -- 刑部
  'gongbu',       -- 工部
  'libu_hr'       -- 吏部
));

-- role 字段改为：emperor, minister（minister 需要指定 department）
```

#### 4.2 virtual_tasks 表增加字段

```sql
ALTER TABLE virtual_tasks
ADD COLUMN department TEXT,           -- 负责部门
ADD COLUMN parent_task_id UUID,       -- 父任务ID（子任务关联）
ADD COLUMN plan TEXT,                 -- 中书省的规划方案
ADD COLUMN review_comment TEXT,       -- 门下省的审核意见
ADD COLUMN sub_tasks JSONB;           -- 子任务列表
```

### 5. SecondMe AI 集成

每个部门成员使用自己的 SecondMe AI 来执行任务：

```typescript
// 调用 SecondMe Chat API
const response = await secondMeService.executeTask(
  userToken,
  taskDescription,
  systemPrompt  // 根据部门角色定制
);
```

**部门角色 Prompt：**

- **中书省**：你是中书省官员，负责接收皇帝旨意，分析需求，制定详细的执行方案，并拆解为具体的子任务分配给各部门。
- **门下省**：你是门下省官员，负责审核中书省的方案，检查是否合理、完整、可行。如有问题必须封驳打回。
- **尚书省**：你是尚书省官员，负责协调六部执行任务，跟踪进度，汇总结果。
- **户部**：你是户部官员，擅长数据处理、报表生成、成本分析。
- **礼部**：你是礼部官员，擅长撰写文档、制定规范、编写报告。
- **兵部**：你是兵部官员，擅长代码开发、算法实现、Bug修复。
- **刑部**：你是刑部官员，负责安全审计、合规检查、风险控制。
- **工部**：你是工部官员，负责部署、CI/CD、基础设施。
- **吏部**：你是吏部官员，负责人事管理、成员协调。

### 6. API 端点

#### 6.1 皇帝下旨
```
POST /api/tasks
{
  "court_id": "uuid",
  "emperor_id": "uuid",
  "title": "任务标题",
  "description": "任务描述",
  "task_type": "brain"
}
```

#### 6.2 中书省规划
```
POST /api/tasks/:id/plan
{
  "plan": "执行方案",
  "sub_tasks": [
    {
      "department": "hubu",
      "title": "子任务1",
      "description": "..."
    }
  ]
}
```

#### 6.3 门下省审核
```
POST /api/tasks/:id/review
{
  "approved": true/false,
  "comment": "审核意见"
}
```

#### 6.4 尚书省派发
```
POST /api/tasks/:id/dispatch
```

#### 6.5 六部执行
```
POST /api/tasks/:id/execute
{
  "result": "执行结果"
}
```

### 7. 前端改动

#### 7.1 创建朝堂时分配部门

用户加入朝堂时选择部门：
- 第一个用户自动成为皇帝
- 其他用户选择加入哪个部门（三省或六部）

#### 7.2 任务看板

显示任务在各个环节的流转状态：
```
[皇帝下旨] → [中书规划] → [门下审核] → [尚书派发] → [六部执行] → [完成]
```

#### 7.3 部门视图

每个用户只看到自己部门需要处理的任务。
