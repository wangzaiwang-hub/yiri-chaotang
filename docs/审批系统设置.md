# 审批系统设置指南

## 数据库迁移

在使用审批系统（结、准、驳）功能之前，需要先执行数据库迁移脚本。

### 步骤：

1. 打开 Supabase Dashboard
2. 进入 SQL Editor
3. 执行以下脚本：`database/add_approval_system.sql`

或者直接复制以下 SQL 并执行：

```sql
-- 添加审批系统相关字段
-- 为 virtual_tasks 表添加审批状态和对话历史

ALTER TABLE virtual_tasks
ADD COLUMN IF NOT EXISTS approval_status VARCHAR(20) DEFAULT 'pending',
ADD COLUMN IF NOT EXISTS conversation_history JSONB DEFAULT '[]'::jsonb,
ADD COLUMN IF NOT EXISTS rejection_count INTEGER DEFAULT 0;

-- 审批状态说明：
-- pending: 待审批（大臣刚回复完）
-- approved: 已准奏（皇帝同意）
-- rejected: 已驳回（皇帝驳回，需要重新设计）
-- closed: 已结束（皇帝选择结束对话）

-- conversation_history 格式：
-- [
--   { "role": "emperor", "content": "任务描述", "timestamp": "2024-01-01T00:00:00Z" },
--   { "role": "minister", "content": "大臣回复", "timestamp": "2024-01-01T00:01:00Z" },
--   { "role": "emperor", "content": "驳回理由", "timestamp": "2024-01-01T00:02:00Z" },
--   { "role": "minister", "content": "重新设计", "timestamp": "2024-01-01T00:03:00Z" }
-- ]

COMMENT ON COLUMN virtual_tasks.approval_status IS '审批状态：pending/approved/rejected/closed';
COMMENT ON COLUMN virtual_tasks.conversation_history IS '完整对话历史记录';
COMMENT ON COLUMN virtual_tasks.rejection_count IS '被驳回次数';
```

## 功能说明

### 三个审批按钮（从左到右）：

1. **结** - 结束对话
   - 点击后，对话结束，可以开启新对话
   - 任务状态变为 `closed`

2. **准** - 准奏（同意方案）
   - 点击后，皇帝同意大臣的方案
   - 任务状态变为 `approved`

3. **驳** - 驳回（方案不行）
   - 点击后弹出输入框，皇帝需要输入驳回理由
   - 系统会将驳回理由发送给大臣的虚拟人
   - 大臣的虚拟人会根据上下文和驳回理由重新设计方案
   - 对话历史会被保存，确保 AI 能理解完整的上下文
   - 驳回次数会累加，怨气值会增加

### 对话流程：

1. 皇帝下圣旨 → 大臣虚拟人回复
2. 皇帝看到回复后，可以选择：
   - **结**：结束对话
   - **准**：同意方案，开始执行
   - **驳**：驳回方案，输入理由 → 大臣重新设计 → 回到步骤 2

### 技术实现：

- 使用 `conversation_history` 字段保存完整对话历史（JSONB 格式）
- 驳回时，将理由追加到对话历史
- 调用 SecondMe API 时，将完整上下文发送给虚拟人
- 虚拟人根据上下文重新生成方案
- 通过 WebSocket 实时推送任务更新

## 注意事项

- 只有皇帝能看到审批按钮
- 审批按钮只在大臣回复完成后显示（打字机效果结束后）
- 驳回时必须输入理由，否则无法提交
- 每次驳回都会增加怨气值
- 对话历史会一直保存，直到任务结束
