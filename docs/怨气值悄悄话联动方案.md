# 怨气值 + 悄悄话联动方案

## 核心理念

**怨气值是大臣的"反叛进度条"，悄悄话是大臣的"密谋工具"**

- 怨气值越高 → 大臣越想反叛
- 悄悄话是大臣积累怨气、策划反叛的地方
- 皇帝可以通过监听悄悄话来发现反叛阴谋

---

## 完整流程

### 第一阶段：怨气积累（0-30分）

**触发场景：**
1. 完成任务被打回：+15 怨气
2. 被虚拟惩罚：+20 怨气
3. 每日签到：+5 怨气
4. 幸福度下降时：+10 怨气

**UI 表现：**
```
怨气值: ███░░░░░░ 30/100
状态: 🟡 不满（可以发送悄悄话）
```

**大臣行为：**
- 可以在悄悄话中吐槽皇帝
- 可以发送"怨言"类型的悄悄话
- 怨气值越高，悄悄话越激进

---

### 第二阶段：密谋阶段（30-70分）

**怨气值达到 30 时解锁：**
- 🔓 解锁"密谋"功能
- 大臣可以在悄悄话中拉拢其他大臣
- 可以发送"密谋"类型的悄悄话

**悄悄话新增类型：**

```typescript
enum WhisperType {
  COMPLAINT = 'complaint',      // 吐槽（0-30分）
  CONSPIRACY = 'conspiracy',    // 密谋（30-70分）
  REBELLION = 'rebellion',      // 反叛（70-100分）
  NORMAL = 'normal'             // 普通聊天
}
```

**密谋悄悄话示例：**
```
大臣A → 大臣B:
"兄弟，你看皇帝这样压榨我们，咱们一起反了他吧？"
[密谋] [拒绝] [同意]

如果大臣B同意：
- 大臣B 怨气值 +10
- 建立"密谋同盟"关系
- 两人的悄悄话会被标记为"密谋中"
```

**皇帝可以看到的信息：**
- 🔴 警告：大臣A 和 大臣B 在密谋
- 建议行动：
  - 🎁 赏赐大臣A（-20怨气）
  - 🛡️ 启动监察（发现密谋内容）
  - ⚖️ 审判大臣A（需要证据）

---

### 第三阶段：反叛准备（70-100分）

**怨气值达到 70 时解锁：**
- 🔓 解锁"起义"功能
- 大臣可以在悄悄话中发起"政变投票"
- 可以发送"反叛"类型的悄悄话

**反叛悄悄话示例：**
```
大臣A → 所有大臣:
"各位，我已经忍无可忍了！谁愿意和我一起推翻这个暴君？"
[发起政变] [支持] [反对] [中立]

政变流程：
1. 发起者需要至少 2 名支持者
2. 进入 24 小时倒计时
3. 其他大臣可以选择立场
4. 皇帝可以采取行动阻止
```

## 数据库设计

### 1. 扩展 court_members 表

```sql
ALTER TABLE court_members ADD COLUMN (
  grudge_value INTEGER DEFAULT 0,           -- 怨气值
  rebellion_status TEXT DEFAULT 'loyal',    -- 忠诚/中立/反叛
  conspiracy_allies TEXT[] DEFAULT '{}',    -- 密谋同盟
  last_grudge_action TIMESTAMPTZ,           -- 最后一次怨气行动
  rebellion_started_at TIMESTAMPTZ          -- 反叛开始时间
);
```

### 2. 新增 whisper_types 表

```sql
CREATE TABLE whisper_types (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  court_id UUID REFERENCES courts(id),
  sender_id UUID REFERENCES users(id),
  recipient_id UUID REFERENCES users(id),
  type TEXT CHECK (type IN ('normal', 'complaint', 'conspiracy', 'rebellion')),
  content TEXT NOT NULL,
  grudge_impact INTEGER DEFAULT 0,          -- 对怨气值的影响
  is_monitored BOOLEAN DEFAULT FALSE,       -- 是否被皇帝监听
  created_at TIMESTAMPTZ DEFAULT NOW()
);
```

### 3. 新增 conspiracies 表

```sql
CREATE TABLE conspiracies (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  court_id UUID REFERENCES courts(id),
  initiator_id UUID REFERENCES users(id),
  status TEXT DEFAULT 'planning',           -- planning/voting/executing/failed
  supporters TEXT[] DEFAULT '{}',           -- 支持者列表
  opponents TEXT[] DEFAULT '{}',            -- 反对者列表
  started_at TIMESTAMPTZ DEFAULT NOW(),
  expires_at TIMESTAMPTZ,                   -- 24小时后过期
  created_at TIMESTAMPTZ DEFAULT NOW()
);
```

## 悄悄话机制详解

### 悄悄话类型与怨气值关联

| 类型 | 怨气范围 | 内容示例 | 皇帝可见 |
|------|---------|---------|---------|
| 普通 | 0-100 | "今天天气真好" | ❌ |
| 吐槽 | 0-30 | "皇帝真烦人" | ⚠️ 监察时可见 |
| 密谋 | 30-70 | "咱们一起反了他" | ⚠️ 监察时可见 |
| 反叛 | 70-100 | "发起政变！" | ✅ 总是可见 |

### 悄悄话对怨气值的影响

```typescript
// 发送悄悄话时
const whisperImpact = {
  normal: 0,           // 普通聊天不影响
  complaint: 2,        // 吐槽 +2 怨气
  conspiracy: 5,       // 密谋 +5 怨气
  rebellion: 10        // 反叛 +10 怨气
};

// 接收悄悄话时
const receiverImpact = {
  normal: 0,
  complaint: 1,        // 听到吐槽 +1 怨气
  conspiracy: 8,       // 被拉拢密谋 +8 怨气
  rebellion: 15        // 被邀请反叛 +15 怨气
};
```

## 完整游戏循环

```
Day 1:
皇帝发布任务 → 大臣完成 → 皇帝打回
↓
大臣怨气 +15 → 发送吐槽悄悄话 → 怨气 +2

Day 2:
大臣怨气 30 → 解锁密谋功能
大臣A 邀请 大臣B 密谋 → 大臣B 怨气 +8
↓
皇帝看到警告 → 启动监察 → 发现密谋
↓
皇帝选择：
  A) 赏赐大臣A (-20怨气) → 大臣A 感激
  B) 拉拢大臣B (-30怨气) → 大臣A 发现背叛 (+50怨气)
  C) 审判大臣A (关天牢) → 其他大臣 +20怨气

Day 3:
大臣A 怨气 70 → 解锁反叛功能
大臣A 发起政变 → 需要 2 名支持者
↓
大臣B, C 同意 → 进入 24 小时投票
↓
皇帝可以：
  A) 收买大臣B (-50幸福度)
  B) 启动重度监察 (-50幸福度)
  C) 妥协谈判 (让出权力)
↓
24 小时后：
  - 政变成功 → 大臣A 成为新皇帝
  - 政变失败 → 所有反叛者关天牢 3 天
```

## 社交互动

### 大臣之间

- 通过悄悄话建立信任
- 共同的怨气值会增加亲密度
- 可以组建派系或同盟
- 互相支持或背叛

### 皇帝与大臣

- 皇帝可以通过赏赐、升官来降低怨气
- 大臣可以通过悄悄话来表达不满
- 形成"压迫-反抗"的动态平衡

---

## 这个方案的优势

1. **完整的故事线**：从不满 → 密谋 → 反叛 → 政变
2. **真实的社交互动**：悄悄话是大臣的真实想法
3. **皇帝的挑战**：需要平衡压迫和安抚
4. **高度的可玩性**：每局游戏都不同
5. **自然的游戏循环**：怨气值驱动所有事件

这样怨气值就不再是孤立的数值，而是整个游戏的核心驱动力！
