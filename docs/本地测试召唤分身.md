# 本地测试召唤分身功能

## 当前配置

已将前端配置为连接本地后端服务器，这样你可以立即测试召唤分身功能。

### 修改的文件

**`frontend/src/services/api.ts`**
```typescript
// 从这个：
const API_BASE_URL = 'https://backend-production-a216.up.railway.app';

// 改为这个：
const API_BASE_URL = 'http://localhost:3001';
```

## 测试步骤

### 1. 确认后端正在运行

后端服务器应该已经在运行（端口 3001）。如果没有运行，执行：

```bash
cd backend
npm run dev
```

### 2. 刷新前端页面

在浏览器中强制刷新前端页面：
- Mac: `Cmd + Shift + R`
- Windows: `Ctrl + Shift + R`

### 3. 测试召唤分身

1. 点击左上角的"召唤分身"按钮（紫色边框）
2. 应该能看到可召唤的分身列表（不再是 404 错误）
3. 点击任意分身进行召唤
4. 确认召唤成功，分身出现在朝堂中

### 4. 检查浏览器控制台

打开开发者工具（F12），确认：
- ✅ 没有 404 错误
- ✅ 没有无限循环错误
- ✅ API 请求成功（状态码 200）

## 测试完成后

### 恢复生产环境配置

测试完成后，需要将前端配置改回生产环境：

**`frontend/src/services/api.ts`**
```typescript
// 改回生产环境
const API_BASE_URL = 'https://backend-production-a216.up.railway.app';
```

1. **恢复配置**

## 本地测试 vs 生产环境

| 环境 | 后端地址 | 前端地址 | 用途 |
|------|---------|---------|------|
| 本地测试 | http://localhost:3001 | http://localhost:3000 | 开发和测试 |
| 生产环境 | https://backend-production-a216.up.railway.app | https://your-frontend.vercel.app | 正式使用 |

## 常见问题

### Q: 刷新后还是 404？

A: 检查以下几点：
1. 后端服务器是否在运行（`npm run dev` 在 backend 目录）
2. 前端是否已经刷新（强制刷新）
3. 浏览器控制台中的请求 URL 是否是 `http://localhost:3001`

### Q: CORS 错误？

A: 本地开发时，后端已经配置了 CORS 允许所有来源，不应该有 CORS 问题。如果有，检查后端配置。

### Q: 如何知道后端正在运行？

A: 访问 http://localhost:3001/health，应该返回：
```json
{
  "status": "ok",
  "timestamp": "..."
}
```

### Q: 测试完成后忘记改回生产环境配置？

A: 如果推送了本地配置到生产环境，前端会尝试连接 `localhost:3001`，导致所有用户都无法使用。记得在推送前检查配置！

## 快速切换脚本

为了方便切换环境，可以使用环境变量：

**`frontend/src/services/api.ts`**
```typescript
const API_BASE_URL = import.meta.env.VITE_API_URL || 'https://backend-production-a216.up.railway.app';
```

**`.env.local`**（本地测试）
```
VITE_API_URL=http://localhost:3001
```

**`.env.production`**（生产环境）
```
VITE_API_URL=https://backend-production-a216.up.railway.app
```

这样就不需要手动修改代码了！
